## 3. Storage Strategy

### Original receipt / raw proof

```yaml
storage: private_disk_or_private_bucket
public_access: false
search_indexing: false
sitemap_llms: false
admin_access: signed_url_or_private_storage_driver
retention: operator_policy_required
```

Examples of acceptable storage patterns:

- Laravel private disk not exposed through public symlink.
- Private S3/R2/Spaces bucket with no public ACL.
- Admin-only signed URL with short TTL.
- Encrypted private archive with backend-authenticated retrieval.

### Redacted public proof

```yaml
storage: public_safe_media_or_public_signed_url
public_access: allowed_after_review
contains_only_redacted_fields: true
proof_public_url: points_to_redacted_proof_only
```

### Prohibited storage patterns

```yaml
forbidden:
  - raw receipt uploaded to public disk
  - raw receipt URL pasted into proof_public_url
  - public proof containing payment account/email/phone/address/full transaction number
  - private proof path returned by public API
  - redaction notes returned by public API
```

## 4. Redaction Standard

The Operator suggested "无需打码"; this SOP does **not** accept that as an absolute rule.

Final standard:

```yaml
no_redaction_allowed_only_if: "reviewer confirms the proof contains no sensitive fields"
redaction_required_if_any_sensitive_field_exists: true
```

### Fields that may remain visible

```yaml
may_remain_visible:
  - recipient_name: United Nations Foundation
  - donation_date
  - donation_amount: CNY 10
  - currency
  - receipt_issuer_name if non-sensitive
  - proof_status
  - short public reference code generated by FermatMind
```

### Fields that must be redacted

```yaml
must_redact:
  - donor full legal name unless operator explicitly approves public display
  - donor email
  - donor phone
  - donor address
  - bank account
  - card number
  - payment account id
  - full transaction number
  - full order number
  - payment provider user id
  - invoice/billing address
  - QR payment token
  - session id
  - auth token
  - receipt private URL
  - browser URL containing private token

```

### Transaction identifier rule

```yaml
transaction_identifier:
  public_display: "forbidden full value"
  allowed_public_form: "optional short masked reference, e.g. last 4-6 chars, only if reviewer approves"
```

## 5. Withheld Proof Policy

`proof_status=withheld` is allowed only when public disclosure would expose sensitive data or when no public-safe proof can be created yet.

Withheld proof records:

- may appear in private ledger;
- may be public only if a reviewer approves a safe explanation;
- must not power trust badge;
- must not be used as high-amplification social proof;
- must preserve private receipt internally.

## 6. Review Workflow

1. Operator makes donation.
2. Save raw receipt to private disk/bucket.
3. Create draft/private DailyGiving record.
4. Create a redacted public proof artifact if possible.
5. Reviewer checks redaction.
6. Reviewer confirms recipient/date/amount.
7. Reviewer confirms forbidden claims absent.
8. If safe, set record toward public eligibility.
9. Keep `is_indexable=false` until indexability gate passes.
10. Do not enable trust badge.

```yaml
    - no official affiliation wording
```

## 8. Public API Rules

Public API must not return:

```yaml
forbidden_public_api_fields:
  - proof_private_path
  - proof_redaction_notes
  - receipt_reference_private
  - internal_notes
  - admin_id
  - raw_transaction_id
  - payment_account
  - private_sync_diagnostics
```

Public API may return after gates:

```yaml
allowed_public_api_fields:
  - record_public_id
  - recipient_name
  - donation_date
  - amount
  - currency
  - proof_status
  - proof_public_url_if_redacted_and_approved
  - public_note_if_claim_linted
```

## 10. Social Sync Rules

Operator may start social video/image updates today, but the posts must be treated as manual social activity, not as evidence that the public ledger or trust badge is live.

Social proof must not include:

- raw receipts;
- payment IDs;
- donor private details;
- full transaction numbers;
- official partnership claims;
- trust badge claims;
- guaranteed impact claims.
